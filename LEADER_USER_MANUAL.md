# 面向领导的系统使用说明

## 1. 项目一句话介绍

本项目是一个案件证据辅助研判 demo。它把笔录、图片证据、研判报告、法医检测报告和法律条文放到同一个分析流程里，由多个 AI 模块分别完成“材料读取、事实提炼、冲突发现、法条关联、报告生成和边界复核”等工作，最后形成一份供办案人员参考的辅助分析报告。

它的目标不是替代民警、检察官、法官作判断，而是帮助工作人员更快发现材料中的关键事实、矛盾点、遗漏点和可能关联的法律依据。

## 2. 这个系统主要解决什么问题

传统案件材料审查中，常见痛点有几个：

第一，材料多。一个案件可能有多份询问笔录、图片辨认材料、现场照片、监控研判报告、法医鉴定报告等，人工逐份阅读容易遗漏。

第二，材料之间可能互相冲突。例如笔录中双方都说“没有打架”，但验伤报告显示存在伤情；或者一份笔录说没有拿手机，另一份图片或报告显示有人拿走手机。

第三，笔录原文很长。真正需要进入分析的往往不是整篇笔录，而是“谁、什么时间、在哪里、做了什么、造成什么后果”这些结构化事实。

第四，法条关联容易被泛化词干扰。例如材料中出现“手机”“财物”，并不一定就是盗窃；如果事实是“摔坏手机”，更可能需要关注故意毁坏财物或其他相关依据。

本系统就是围绕这些问题设计的：先把材料转成清晰事实，再做冲突检查和法律依据关联，最后生成辅助报告。

## 3. 总体工作流程

系统的基本流程可以理解为八步：

1. 用户把案件材料放进指定证据文件夹。
2. 系统先盘点材料数量和类型。
3. 人工确认案件类型，例如“故意伤害类案件”。
4. 各个 AI 模块分别处理笔录、图片、报告等材料。
5. 系统把各类材料提炼成统一的案件事实图谱。
6. 系统检查不同材料之间是否存在冲突。
7. 系统根据事实匹配可能相关的静态法律条文。
8. 系统生成辅助分析报告，并由复核模块检查是否越界。

这里特别强调一点：系统必须经过人工确认案件类型后才开始正式分析。也就是说，AI 不会绕过人工案件定性直接进入推理。

## 4. 各模块分别做什么

### 4.1 材料接收模块：Evidence Intake

这个模块负责把用户放入证据文件夹的材料读进系统。

它支持三类主要材料：

- 笔录材料：可以是 Word、PDF 或文本文件。
- 图片证据：例如现场照片、图片辨认、证据照片。
- 报告材料：例如法医检测报告、监控研判报告，可以是图片、Word 或 PDF。

它的作用类似“材料收发员”。它不做判断，只负责识别材料类型、读取材料内容、生成材料清单，并把材料交给后续模块处理。

如果 PDF 是扫描版，系统本地不会做 OCR。图片和扫描材料的理解交给 Qwen 视觉模型；如果识别结果需要人工修正，可以在指定位置放同名文本，系统会优先使用人工修正内容。

### 4.2 规划模块：Planning Agent

Planning Agent 负责开工前的“材料盘点”和“任务分配”。

它会先看当前证据文件夹里有多少份笔录、多少组图片证据、多少份报告材料，然后形成一个处理计划。

它还可以根据材料内容提出案件类型建议，但正式运行前必须由人工确认案件类型。比如系统可以提示“可能是故意伤害类案件”，但必须由使用者明确输入或确认后，后续模块才会继续工作。

这个设计是为了避免 AI 自行定性案件，保证关键判断仍由人控制。

### 4.3 笔录分析模块：Text Agent

Text Agent 负责阅读笔录。

它不是把整篇笔录复制进数据库，而是把笔录提炼成短事实。系统重点提取：

- 谁说的；
- 谁做了什么；
- 事情发生在什么时间；
- 发生在什么地点；
- 涉及什么对象、伤情、物品或后果；
- 是否存在否认类陈述。

例如笔录原文可能很长：

```text
2026年6月12日，在深圳市宝安区新凯飞汽配，我和同事李文杰在工作群里因为工作问题发生口角，之后他来我工位把我的手机摔在地上，导致屏幕损坏。
```

Text Agent 会把它整理为：

```text
李文杰摔坏手机并造成手机损坏
时间：2026年6月12日
地点：深圳市宝安区新凯飞汽配
对象：手机
```

如果笔录中有人说“没有打架”“没有动手”“没有拿手机”“没有损坏财物”，Text Agent 也会把这些否认内容单独提取出来。这样后续冲突检测模块才能发现：一边说没有发生，另一边报告或图片却显示发生了。

### 4.4 图片证据分析模块：Pic Agent

Pic Agent 负责处理现场照片、图片辨认、证据照片等图片材料。

它会调用 Qwen 视觉模型，让模型描述图片中看到了什么，并识别图片中可见的文字。然后系统再把图片描述提炼成案件事实。

例如图片可能显示：

- 某人在某个位置出现；
- 某个物品损坏；
- 某个现场状态；
- 图片下方有签字确认内容；
- 辨认照片中标注了某个人或某个物品。

Pic Agent 的作用是把“图片里有什么”转成可以参与分析的文字事实。

如果同一件事有多张图片，用户可以把它们放在同一个文件夹中。系统会把同一个文件夹作为一组图片处理，避免把不同事件的图片混在一起分析。

### 4.5 报告材料分析模块：Report Image Agent

Report Image Agent 负责处理法医检测报告、监控研判报告等报告类材料。

虽然名字里有 Image，但现在它不只处理图片，也支持 Word 和 PDF 报告。

它主要提取报告里的结论性事实，例如：

- 法医报告中认定的伤情；
- 被鉴定人是谁；
- 鉴定意见是什么；
- 监控研判报告显示谁在什么时间出现；
- 报告是否提到某个行为或后果。

例如法医报告中写明“被鉴定人贺显作所受损伤为轻伤二级”，系统会把它提炼为一条报告事实，供后续模块与笔录说法对照。

这类报告材料通常对冲突检测很重要，因为它们可以和当事人口供互相印证或互相矛盾。

### 4.6 案件事实图谱模块：Evidence Graph Agent

Evidence Graph Agent 可以理解为“案件事实整理员”。

前面的 Text Agent、Pic Agent、Report Image Agent 都会产出各自的事实。Evidence Graph Agent 把这些事实统一合并成一个案件事实图谱，也叫 Case Graph。

目前这个图谱是一个轻量版事实图谱，不是复杂图数据库。它的核心是很多条结构化事实，每条事实都包含：

- 来源材料；
- 涉及人员；
- 行为；
- 时间；
- 地点；
- 对象或后果；
- 置信度。

这个模块的意义是：后面的冲突检测、法条匹配、报告生成，都不直接面对杂乱原文，而是面对已经整理好的事实。

这能减少上下文混乱，也能让报告中的每个判断尽量追溯到具体材料。

### 4.7 冲突检测模块：Conflict Agent

Conflict Agent 负责找材料之间的矛盾。

它不是只针对某一种案件，而是按照通用冲突类型检查，例如：

- 一方否认打架，但报告显示有人受伤；
- 一方否认动手，但图片或报告显示存在肢体冲突；
- 一方否认拿取物品，但图片或报告显示物品被拿走；
- 一方否认损坏财物，但现场照片或笔录显示财物损坏；
- 多份材料对时间、地点、行为人、对象描述不一致。

例如，笔录里双方都说“没有打架”，但法医检测报告显示有轻伤结果，Conflict Agent 就应该提示：口供与客观报告之间存在需要核查的冲突。

这个模块的价值在于提前暴露疑点，提醒工作人员重点复核，而不是等到报告末尾才发现材料互相打架。

### 4.8 法律依据检索工具：Legal Retrieval Tool

Legal Retrieval Tool 负责根据案件事实查找可能相关的法律条文。

当前 demo 使用的是静态法律库，也就是项目中的 `legal_library/laws.jsonl` 文件。它不是联网搜索，也不是完整法律数据库，而是一个可人工维护、可审计的法条库。

它会根据案件类型、行为事实、关键词和法律构成要素匹配可能相关的法条。

系统现在特别加强了一个规则：不能因为材料中出现“手机”“财物”就直接匹配盗窃条款。比如“李文杰摔坏手机”更接近财物损坏事实，不应仅因出现“手机”就关联盗窃罪。

因此，盗窃条款通常需要出现更强的盗窃语义，例如“盗窃、偷、窃取、拿走、非法占有”等。

这个工具输出的是“可能关联的法律依据”，不是最终法律定性。

### 4.9 推理报告模块：Reasoning Agent

Reasoning Agent 负责把前面整理出的事实、冲突点和法律依据组织成一份辅助分析报告。

它的输入主要包括：

- 案件事实图谱；
- 冲突检测结果；
- 可能关联的法律条文；
- 人工确认的案件类型。

它会围绕这些内容进行整理：

- 现有证据显示了什么行为；
- 谁可能在案件中扮演什么角色；
- 哪些事实有材料支持；
- 哪些事实之间存在矛盾；
- 可能关联哪些法律依据；
- 哪些地方需要人工进一步核实。

它不能凭空猜测，也不能脱离材料作判断。如果材料没有显示某个事实，报告中不应把它写成确定结论。

### 4.10 反方质询模块：Judge Agent

Judge Agent 不是最终裁判，也不是作决定的人。

它的角色更像“反方审查员”或“陪审团中的反方意见”。它专门挑战 Reasoning Agent 的报告，看报告中有没有这些问题：

- 证据支持是否不足；
- 是否忽略了冲突；
- 是否把不确定事实写得过于确定；
- 是否引用了不够相关的法条；
- 是否遗漏了关键疑点；
- 是否需要重新表述或补充说明。

这个模块的意义是防止系统自己生成一份看似完整但没有经过挑战的报告。它会主动提出质疑，让报告更谨慎。

### 4.11 边界复核模块：Review Agent

Review Agent 是最后一道安全边界。

它主要检查最终报告是否越界，例如：

- 是否把辅助分析写成了最终定罪；
- 是否写了“已经构成犯罪”“应当判处”等最终性结论；
- 是否引用了没有来源的法律依据；
- 是否使用了 Case Graph 中没有的事实；
- 是否把推测写成确定事实。

如果发现报告越界，Review Agent 会提示问题，要求修改。

这个模块体现的是系统定位：它只能做辅助研判，不能替代人作法律判断。

## 5. 一份材料进入系统后会发生什么

可以用一个简化例子说明。

假设用户放入这些材料：

- 贺显作的询问笔录；
- 李文杰的询问笔录；
- 一份法医检测报告；
- 一组现场或监控图片；
- 静态法律库中的相关法律条文。

系统运行后：

1. Evidence Intake 读取材料，识别哪些是笔录、哪些是图片、哪些是报告。
2. Planning Agent 盘点材料，提示本次有几份笔录、几组图片、几份报告。
3. 用户确认案件类型为“故意伤害类案件”。
4. Text Agent 分别读取每份笔录，提炼出短事实，例如“李文杰摔坏手机”“李文杰称没有打架”。
5. Pic Agent 识别图片，看图片中是否有现场状态、人物、物品、文字或签名。
6. Report Image Agent 读取法医报告或研判报告，提炼出伤情结论或监控结论。
7. Evidence Graph Agent 把所有事实放入统一的案件事实图谱。
8. Conflict Agent 检查是否存在“口供否认但报告显示相反情况”的冲突。
9. Legal Retrieval Tool 查找可能相关的法律依据。
10. Reasoning Agent 生成辅助分析报告。
11. Judge Agent 对报告提出质疑。
12. Reasoning Agent 根据质疑修订报告。
13. Review Agent 检查报告是否超出证据和法律边界。

最终输出的是一份辅助分析报告，而不是最终裁判意见。

## 6. 输出报告应该怎么看

系统输出报告时，建议重点看四类内容。

第一，看行为事实。也就是系统认为现有材料中明确显示了哪些行为。例如谁摔坏了什么、谁否认了什么、报告显示了什么伤情。

第二，看时间地点。系统会尽量整理出事情发生的时间和地点，方便人工核对材料是否一致。

第三，看冲突与纰漏。这里是报告最有价值的部分之一，能够提示哪些材料之间不一致，哪些事实需要继续核查。

第四，看可能关联的法律依据。这里不是最终定性，只是提示哪些法条可能需要人工进一步审查。

如果报告中出现“不确定”“需人工核实”“存在冲突”等表述，不代表系统失败，而是说明系统发现了材料中的不稳定点。这些内容反而应该成为人工复核重点。

## 7. 这个系统的边界

本系统有明确边界：

第一，它不替代办案人员判断。系统只能辅助整理、提示和分析，不能决定案件性质，也不能作最终法律结论。

第二，它不保证材料真实。系统只能基于输入材料进行分析，材料本身是否真实、完整、合法，需要人工审查。

第三，它不自动联网找法条。当前 demo 使用静态法律库，需要人工维护法律条文。

第四，它不在本地做 OCR。图片识别依赖外部 Qwen 视觉服务，PDF 扫描件需要外部识别或人工提供文本。

第五，它不能凭空补事实。材料中没有明确显示的内容，系统不应写成确定事实。

这些边界是故意保留的，目的是让系统可控、可解释、便于人工复核。

## 8. 为什么采用多 Agent 协同

本项目没有让一个大模型从头到尾直接生成报告，而是拆成多个模块，是为了降低风险。

每个模块只做一类工作：

- 读笔录的只读笔录；
- 看图片的只看图片；
- 管事实图谱的只整理事实；
- 查冲突的只查冲突；
- 找法条的只找可能依据；
- 写报告的只基于整理后的事实写报告；
- 质询和复核模块负责检查报告是否稳妥。

这种设计有三个好处。

第一，职责清楚。哪个环节出了问题，比较容易定位。

第二，减少混乱。每份笔录、每组图片都尽量独立处理，避免不同材料互相污染。

第三，更容易人工监管。关键节点，例如案件类型确认、冲突复核、法条维护，都保留人工控制。

## 9. 当前 demo 已实现的重点能力

当前版本已经具备以下能力：

- 支持证据文件夹统一收集材料；
- 支持笔录 Word、PDF、文本读取；
- 支持报告图片、Word、PDF 作为报告类证据；
- 支持 Qwen 视觉模型描述图片证据；
- 支持 DeepSeek 文本模型参与文本分析；
- 支持单份笔录独立上下文提取；
- 支持同一组图片独立上下文识别；
- 支持把材料提炼为结构化案件事实；
- 支持检查口供、图片、报告之间的冲突；
- 支持静态法律库匹配可能关联法条；
- 支持反方质询和最终边界复核。

这些能力已经可以支撑 demo 级演示：把真实材料放入证据文件夹，人工确认案件类型，系统输出一份结构化辅助研判报告。

## 10. 后续可扩展方向

如果后续继续建设，可以从几个方向升级。

第一，升级法律库。当前是静态法律库，后续可以接入更完整的法律法规、司法解释、指导案例和本地规范文件。

第二，升级知识图谱。目前 Case Graph 是轻量事实图谱，后续可以扩展为人员、地点、物品、行为、伤情、证据之间的关系网络。

第三，升级人工复核界面。未来可以让工作人员在界面上确认或修改每条事实，而不是只看文本报告。

第四，升级冲突检测。可以进一步识别时间线冲突、人员身份冲突、证据来源冲突、证据链缺口等更复杂问题。

第五，升级报告模板。可以按公安、检察、纪检、法务等不同场景生成不同格式的研判报告。

第六，升级权限和留痕。真实使用时，需要记录谁上传材料、谁确认案件类型、谁修改事实、谁采纳报告意见。

## 11. 汇报时可以强调的价值

对领导汇报时，可以概括为四句话：

第一，它能把分散的案件材料变成结构化事实，降低人工阅读压力。

第二，它能主动发现笔录、图片、报告之间的冲突和纰漏，帮助工作人员抓住复核重点。

第三，它能根据事实关联可能相关的法律依据，但不替代人工定性。

第四，它通过 Planning、Conflict、Judge、Review 等模块保留了人工确认和边界复核，整体设计偏谨慎、可控、可追溯。

## 12. 结论

这个 demo 的核心价值不是“让 AI 直接办案”，而是让 AI 做材料整理、疑点提示、辅助分析和报告初稿工作。

它适合用于案件材料初筛、证据矛盾提示、法律依据辅助查找和研判报告草拟。真正的案件判断、法律定性和处理决定，仍然必须由具备权限和专业能力的人员完成。
